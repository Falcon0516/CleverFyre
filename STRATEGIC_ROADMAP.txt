SECTION 1 — THE CORE INSIGHT (YOUR MOAT)
─────────────────────────────────────────────────────────────────────────────

Current AI agent payment systems answer: WHO paid WHAT to WHOM.
x402 Sentinel answers:  WHO paid WHAT to WHOM, and WHY, and was that WHY
                         LEGITIMATE given the agent's assigned mission?

This is called CAUSAL PAYMENT AUDITING. It does not exist anywhere.

Real-world analogy:
  A corporate credit card records the transaction.
  An expense report records the justification.
  Nobody has built the expense report layer for AI agents — until now.

On Algorand, every payment transaction carries an 8-byte note field.
We use this to commit a SHA-256 "Intent Hash" — a cryptographic fingerprint
of the agent's decision context that triggered the payment.

The full decision context (tool call history, task objective, reasoning chain)
is stored off-chain on IPFS. The CID is what gets hashed.

Result: You can reconstruct WHY any agent made any payment, forever,
        from on-chain data alone. Immutable. Tamper-proof. Auditable.

No other tool does this. No other team will think of this.


SECTION 2 — THE 5 FEATURES NOBODY ELSE WILL BUILD
─────────────────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────────────┐
  │  FEATURE 1: CAUSAL INTENT LEDGER                                    │
  │  Status: Does not exist anywhere in production or research          │
  └─────────────────────────────────────────────────────────────────────┘

  What it is:
    Before any payment executes, the wrapper snapshots the agent's current
    decision context: task goal, tool calls made so far, why this API was
    chosen, what the expected output is. This snapshot is serialized as JSON,
    uploaded to IPFS, and the CID is hashed. The hash goes into the Algorand
    tx note field alongside the payment.

  Why it's unique:
    You get a CAUSAL GRAPH of agent behavior. Each payment node links back
    to the task node that spawned it. You can answer: "At 2:47 AM, agent_007
    spent $4.20 on the OpenWeather API. Why? Because it was executing task
    'plan outdoor event for client X' and needed real-time weather data."
    This is forensic-grade AI auditing.

  Technical implementation:
    - snapshot = {task_id, goal, tool_history, api_rationale, expected_output}
    - cid = await ipfs_upload(json.dumps(snapshot))
    - intent_hash = sha256(cid)[:8]  # 8 bytes fits Algorand note field
    - tx.note = f"x402:sentinel:{intent_hash}"
    - Store full CID in Algorand box storage keyed by tx_id
    - Later: anyone can retrieve CID from box, fetch from IPFS, verify hash

  Algorand tools: AlgoKit box storage, AlgorandClient.send_payment()

  ──────────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────────────┐
  │  FEATURE 2: DEAD MAN'S SWITCH POLICY RENEWAL                       │
  │  Status: Does not exist anywhere                                    │
  └─────────────────────────────────────────────────────────────────────┘

  What it is:
    Every agent's spending policy has a "proof of life" expiry. The human
    operator MUST re-sign the policy every N hours (configurable: 6h, 12h,
    24h) by submitting a renewal transaction. If the operator goes silent —
    compromised account, emergency, forgot — the policy auto-expires and ALL
    agent payments FREEZE instantly at the smart contract level.

    This is a Dead Man's Switch for AI agents.

  Why it's unique:
    Every other system assumes the admin is always available and always
    trusted. This system assumes compromise is inevitable and builds
    time-bound trust as the primitive. This is how military systems work.
    No hackathon team in history has applied this to AI payments.

  Why it matters in production:
    In 2025, multiple AI agent incidents involved agents that continued
    operating after their operators had lost control of the system.
    A Dead Man's Switch would have stopped every single one of them.

  Technical implementation:
    AlgoKit contract: PolicyVault.py
    - policy_expiry: UInt64  (stored as Algorand round number)
    - renewal_window: UInt64  (e.g., 14400 rounds ≈ 24 hours at 4s/block)
    - @arc4.abimethod renew_policy(auth_sig): resets expiry to now + window
    - Every payment call checks: Global.round < policy_expiry, else REJECT
    - Renewal tx requires operator's Ed25519 signature (not just sender addr)
    - Emit PolicyRenewed / PolicyExpired events for dashboard

  Frontend shows: A countdown timer per agent. Red when < 2 hours remain.

  ──────────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────────────┐
  │  FEATURE 3: SEMANTIC BUDGET ALLOCATION VIA EMBEDDING POLICY         │
  │  Status: Completely novel — no prior art                            │
  └─────────────────────────────────────────────────────────────────────┘

  What it is:
    Instead of whitelisting specific API URLs (brittle, hardcoded), we define
    budgets by SEMANTIC CATEGORY using sentence embedding vectors.

    The policy says: "This agent can spend up to $10/day on APIs that are
    semantically similar to 'weather data retrieval' and up to $5/day on APIs
    semantically similar to 'geolocation services'."

    When the agent tries to pay for a new API it has never seen before, the
    system computes the embedding of the API's description (from its OpenAPI
    spec or x402 metadata), measures cosine similarity to the policy vectors,
    and routes the spend to the appropriate category budget — automatically.

  Why it's unique:
    No other system on earth does semantic-aware budget allocation. Every
    existing system uses exact URL matching or static category lists.
    This handles the real-world problem: AI agents discover new APIs
    dynamically. You cannot hardcode every API URL in advance.

  Technical implementation:
    - Policy stores: [(category_label, embedding_vector_bytes, budget_usd)]
    - Embeddings are 384-dim MiniLM vectors, quantized to int8 (150 bytes)
    - Stored in Algorand box storage (supports up to 32KB per box)
    - Wrapper SDK: on new API, call embed(api_description) locally
    - Compute cosine similarity against all category vectors
    - Route spend to highest-similarity category if similarity > threshold
    - If no category matches (similarity < 0.7): PAYMENT BLOCKED
    - All similarity computations happen OFF-chain in the SDK (Algorand
      cannot run ML, but it ENFORCES the budget via the spend counter)

  Model: all-MiniLM-L6-v2 via sentence-transformers (30MB, runs locally)
  No API call needed — pure local inference, zero latency overhead.

  ──────────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────────────┐
  │  FEATURE 4: PROMPT INJECTION PAYMENT HONEYPOT DETECTOR              │
  │  Status: Does not exist — security research, not yet productized    │
  └─────────────────────────────────────────────────────────────────────┘

  What it is:
    This is a security feature against a real attack vector:
    A malicious API response injects instructions into the agent's context
    window, causing the agent to make unauthorized payments to attacker-
    controlled endpoints. This is called "Indirect Prompt Injection" and
    it is the #1 security threat to autonomous AI agents in 2025-2026.

    x402 Sentinel detects this by maintaining a behavioral baseline for each
    agent and flagging statistical anomalies in payment patterns.

    Baseline metrics tracked (all stored on-chain via Algorand box storage):
      - rolling_avg_payment_usd (exponential moving average)
      - payment_frequency_per_hour
      - domain_entropy (Shannon entropy of API domain distribution)
      - semantic_drift (cosine distance from historical category centroid)

    If any metric deviates beyond 2.5 standard deviations from baseline,
    the payment is QUARANTINED: held in escrow, operator alerted, agent
    paused until human reviews.

  Why it's unique:
    No payment wrapper — x402 or otherwise — has anomaly detection. They all
    assume the agent's payment intent is always legitimate. We assume it can
    be hijacked and we defend against it.

  Technical implementation:
    - PaymentAnomalyDetector class in SDK (pure Python, numpy only)
    - Updates baseline after each SUCCESSFUL payment
    - Quarantine = SLA Escrow hold with operator_review flag set to True
    - Algorand event: PaymentQuarantined(agent, amount, reason_hash)
    - Dashboard: quarantine queue with approve/reject buttons

  Demo moment: Show a simulated prompt injection → payment blocked in <200ms

SECTION 6 — WHAT MAKES THIS PRODUCTION-GRADE
─────────────────────────────────────────────────────────────────────────────

  Most hackathon projects are toys. Here is why ours is not:

  1. INSTALLABLE SDK — pip install x402-sentinel
     Any AI agent framework (LangChain, AutoGen, CrewAI) can import
     SentinelWrapper in 2 lines. This is a real developer product.

  2. OFFLINE INTENT HASHING — IPFS upload is async, non-blocking.
     Payment does not wait for IPFS. Intent hash is computed locally
     first and committed immediately. IPFS upload resolves in background.
     Zero latency overhead on the payment path.

  3. LOCAL EMBEDDING — MiniLM runs fully locally. No API key.
     No rate limit. No vendor dependency. Semantic routing works offline.

  4. GRACEFUL DEGRADATION — if Algorand localnet is unreachable during
     demo, SDK falls back to an in-memory policy check and logs to file.
     Demo never crashes.

  5. CONTRACT UPGRADABILITY — PolicyVault uses AlgoKit's ARC-4 box
     storage for all mutable state. Contract logic is immutable but
     state is upgradable via admin methods. This is production pattern.

  6. NO VENDOR LOCK-IN — the wrapper is protocol-agnostic. It works with
     any API that returns a standard 402 + x-payment header. Not just
     Algorand APIs. Any HTTP service.
