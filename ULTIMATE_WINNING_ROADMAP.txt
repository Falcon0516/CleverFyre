  SECTION 2 — THE 7 FEATURES (EACH ONE IS A WOW FACTOR)
================================================================================

  ┌─────────────────────────────────────────────────────────────┐
  │  FEATURE 1: AGENT REPUTATION PASSPORT (Soul-Bound Token)   │
  │  Novelty: ★★★★★ — Does not exist anywhere                 │
  └─────────────────────────────────────────────────────────────┘

  What it is:
    Every AI agent that registers with x402 CORTEX receives an
    on-chain Soul-Bound Token (SBT) on Algorand. This SBT is
    non-transferable and accumulates a REPUTATION SCORE based on:
      - payment_reliability: % of payments completed without dispute
      - response_utilization: did the agent actually USE the API response?
      - spending_consistency: how predictable is payment behavior?
      - delegation_trust: has this agent been trusted by other agents?
      - dispute_rate: how often do payments get contested?

    The SBT score is updated after EVERY payment via AlgoKit box storage.
    Services can query any agent's reputation before serving it.
    Higher reputation = lower collateral requirements, priority access,
    and eligibility for cross-agent delegation.

  Why nobody else builds this:
    Other teams see the agent as a wallet. We see it as an identity.
    Credit scores for AI is a concept from 2028 research papers —
    we're building it in 2026 at a hackathon. Judges will remember this.

  Technical implementation:
    - AlgoKit ARC-4 contract: AgentPassport.py
    - Box storage: agent_addr → ReputationData struct
    - ReputationData = {reliability, utilization, consistency,
                        delegation_score, dispute_rate, total_txns,
                        created_round, last_updated_round}
    - Update method: called by CORTEX SDK after each payment cycle
    - Query method: any service can read any agent's reputation
    - SBT minted as ARC-19 ASA with metadata URL pointing to box data

  ────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────┐
  │  FEATURE 2: CAUSAL PAYMENT GRAPH (On-Chain Forensics)      │
  │  Novelty: ★★★★★ — No prior art in production              │
  └─────────────────────────────────────────────────────────────┘

  What it is:
    Before any payment, the SDK captures the FULL decision context:
    task goal, tool history, why this API was chosen, expected output.
    This is serialized, hashed (SHA-256), and the 8-byte hash is
    committed to the Algorand tx note field. Full context is stored
    in AlgoKit box storage keyed by tx_id.

    Result: A directed acyclic graph (DAG) of agent behavior where
    each payment node links to the task node that spawned it.
    Anyone can reconstruct WHY any payment happened, from chain data.

  Implementation:
    - IntentRegistry.py (AlgoKit ARC-4)
    - Box: tx_id → {task_goal, api_url, rationale, expected_output,
                     parent_task_id, timestamp, agent_addr}
    - tx.note = "x402:cortex:" + sha256(context_json)[:8]
    - SDK captures context automatically via decorator pattern

  ────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────┐
  │  FEATURE 3: CROSS-AGENT PAYMENT DELEGATION MESH            │
  │  Novelty: ★★★★★ — Nobody has multi-agent payment delegation│
  └─────────────────────────────────────────────────────────────┘

  What it is:
    Agent A spawns sub-agents B and C for parallel subtasks.
    A DELEGATES payment authority to B and C with constraints:
      - max_spend_per_tx: 0.5 ALGO
      - max_total_spend: 2.0 ALGO
      - allowed_domains: ["api.weather.com", "api.news.org"]
      - expires_at: current_round + 1000

    All payments by B and C link back to A's master task.
    If B exceeds budget or pays an unauthorized domain, A is
    notified and B's delegation is revoked ON-CHAIN.

    This creates a TREE of trust — a delegation DAG stored on
    Algorand — that mirrors how real organizations delegate
    spending authority. No other wrapper has this concept.

  Why this is production-grade:
    In real agentic workflows (AutoGen, CrewAI, LangGraph), agents
    spawn sub-agents all the time. But NONE of them have a payment
    delegation model. They all share one wallet. That's insecure.
    We solve this with cryptographic delegation on Algorand.

  Implementation:
    - DelegationRegistry.py (AlgoKit ARC-4)
    - Box: delegation_id → {parent_agent, child_agent, constraints,
                            spent_so_far, status, created_round}
    - Methods: delegate(), revoke(), check_authority(), spend_against()
    - Atomic: delegation check + payment happen in same tx group

  ────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────┐
  │  FEATURE 4: PREDICTIVE PAYMENT PRE-CACHING                 │
  │  Novelty: ★★★★☆ — Novel optimization, not just safety     │
  └─────────────────────────────────────────────────────────────┘

  What it is:
    Before executing a task, the CORTEX agent analyzes the task
    description and PREDICTS which APIs will be needed.
    It pre-authorizes payments for all predicted APIs in a SINGLE
    Algorand atomic transaction group.

    When the agent actually needs to call an API, the pre-auth
    is already on-chain — payment confirmation is INSTANT.
    Unused pre-auths are released back after task completion.

    This reduces per-call latency from 4+ seconds to ~0 seconds.

  Why judges love this:
    Every other team's demo will have a visible 4-second delay per
    payment (Algorand block time). Our demo will be INSTANT because
    payments were pre-cached. Judges will ask "how is it so fast?"
    That's the conversation where you explain predictive pre-caching.

  Implementation:
    - PreAuthVault.py (AlgoKit ARC-4)
    - Methods: batch_preauth(agent, amounts[], domains[], ttl),
               claim_preauth(agent, domain), release_unused(agent)
    - Uses Algorand AtomicTransactionComposer for batch pre-auth
    - SDK: LLM call to predict needed APIs based on task description
    - Fallback: if prediction misses, standard 402 flow kicks in

  ────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────┐
  │  FEATURE 5: ADVERSARIAL PAYMENT CANARY                     │
  │  Novelty: ★★★★★ — Security primitive that doesn't exist    │
  └─────────────────────────────────────────────────────────────┘

  What it is:
    Before paying full price for an API call, the SDK sends a
    small "canary" payment (1% of price) with a KNOWN test query.
    If the API returns a valid response to the canary, CORTEX
    proceeds with the full payment. If the canary detects:
      - Empty/garbage response
      - Response inconsistent with API description
      - Latency > 10x expected
    → Full payment is BLOCKED. Agent reputation of the SERVICE
      is downgraded. Incident is logged on-chain.

    This protects agents from MALICIOUS x402 services that
    take payment but return trash — the "rug pull" of API economy.

  Why nobody else thinks of this:
    Every other wrapper trusts the server. We verify the server
    BEFORE paying full price. This is "zero trust for APIs."

  Implementation:
    - Canary logic in SDK (Python): canary_verifier.py
    - Sends micro-payment → checks response quality
    - Quality check: response length > 0, valid JSON/text,
      latency within bounds, semantic relevance score
    - On-chain: CanaryLog box stores (domain → canary_results[])
    - Failed canary → domain added to agent's local blocklist

  ────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────┐
  │  FEATURE 6: DEAD MAN'S SWITCH + POLICY GOVERNANCE          │
  │  Novelty: ★★★★☆ — Military-grade human oversight           │
  └─────────────────────────────────────────────────────────────┘

  What it is:
    Every agent's spending policy has a time-bound expiry.
    The human operator MUST re-sign the policy every N hours.
    If operator goes silent → ALL agent payments FREEZE instantly.

    Added twist (beyond the standard concept):
    - Multi-sig governance: 2-of-3 humans can override any freeze
    - Emergency kill switch: any single admin can freeze ALL agents
    - Graduated unfreezing: after freeze, agent resumes at 10% budget
      and scales back up over 1 hour (prevents burst spending on resume)

  Implementation:
    - PolicyVault.py (AlgoKit ARC-4)
    - Fields: policy_expiry, renewal_window, admin_addrs[3],
              emergency_frozen, graduated_resume_start
    - Methods: renew_policy(), emergency_freeze(), unfreeze_graduated()
    - Every payment checks: !frozen AND round < expiry
